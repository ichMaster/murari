# murari — агент брейншторму (концепт прототипу)

Назва проекту: **murari** (санскр. मुरारि — «ворог Мури», епітет Крішни, що здолав демона Муру;
пасує агенту, чия робота — валити слабкі гіпотези).

---

## Суть

Окрема мінімальна аплікація: чат на Haiku з TUI-інтерфейсом, з якого
викликається **агент брейншторму** —
Claude Code сабагент у headless-режимі (`claude -p`). Агент веде цикл
**diverge → select → verify → synthesize** над темою: породжує гіпотези, перевіряє їх у живому вебі
й зі знайденого народжує свіжі ідеї.

Сесія — це робота над документом. Її результат — **робочий документ**: зв'язний текст з ідеями,
підкріпленими перевіреними гіпотезами і джерелами. Він росте протягом сесії, видимий у TUI поруч
із чатом і ledger'ом, і лишається після виходу — це те, заради чого сесія була. Ledger та решта
файлів — робочі матеріали; пам'яті між сесіями немає, кожна нова сесія починає свій документ
з чистого аркуша.

Мета прототипу — **сам агент**: довести, що цикл дає ідеї, простежувані до знахідок пошуку, а не
переказ пріорів моделі чи слів користувача. Патерн kiln: спершу випал тут, потім перенесення в Lumi.

---

## Цілі

- Агент, що за прогін повертає гіпотези з вердиктами (`confirmed` / `refuted` / `partial` + джерело)
  і щонайменше одну ідею з позначкою `born_from: search`.
- Сесія завершується документом: `DOCUMENT.md`, зв'язний текст, де вагомі тези мають джерела.
- Керованість: репліка користувача в чаті змінює напрям наступного прогону і форму документа.
- Бюджети: прогони на сесію, `--max-turns` на прогін, витрати передбачувані.

---

## Архітектура — дві голови, як у kiln

```
ти ──TUI──> haiku loop ──spawn──> claude -p (brainstormer)
             ^                          |
             |     workspace сесії      |
             +-- TOPIC.md   LEDGER.md --+
                 SOURCES.md IDEAS.md
                 DOCUMENT.md   <- результат
```

**Чат-шар** — Python, Haiku по HTTP API. Ролі: розмова, вхід у режим, формулювання теми,
перетворення твоїх реплік на сіди для агента, подача результатів прогону людською мовою. Єдиний
тул Haiku — `run_brainstorm(seed)`, що запускає прогін; жодного іншого доступу (див. «Пісочниця
та ізоляція виклику»).

**Інтерфейс** — TUI на Textual, три панелі: **чат**; **ledger** — гіпотези зі статусами й логом
перевірок по кожній; **робочий документ**. Статус-бар агента (вільний / копає / прогонів
лишилось). Панелі ledger і документа перечитують файли workspace по завершенню прогону; Textual
асинхронний, тож чат живий, поки агент копає. Розкладка [П]: чат зліва, правий стовп поділений —
ledger згори, документ знизу.

**Агент** — Claude Code headless: `claude -p` з означенням `.claude/agents/brainstormer.md`.
Tools обмежені: `WebSearch`, `WebFetch`, `Read`, `Write` (тільки workspace). Модель — Opus 4.8.
Вихід — `--output-format json` за фіксованою схемою.

**Workspace сесії** — тека `MURARI_HOME/sessions/<дата>-<slug теми>/` [П]: `TOPIC.md`,
`LEDGER.md`, `SOURCES.md`, `IDEAS.md`, `DOCUMENT.md`. Агент сам читає і веде ці файли між
прогонами — це його природний спосіб тримати стан; чат-шар їх лише читає. Тека лишається після
виходу: `DOCUMENT.md` — результат, решта — робочі матеріали поруч.

## Цикл агента (ядро)

Один прогін:

1. **read** — `TOPIC.md` + `LEDGER.md` + свіжий сід (остання змістовна репліка користувача).
2. **diverge** — 2–4 нові гіпотези чи кути; закриті в ledger не повторювати.
3. **select** — з них перевірні: мають фактичне ядро і статус `open`.
4. **verify** — `WebSearch` / `WebFetch`; кожній вибраній — вердикт `confirmed` / `refuted` /
   `partial` з джерелом (url). Джерела нема — гіпотеза чесно лишається `open`.
5. **synthesize** — свіжі ідеї, народжені зі знайденого (`born_from: search`, з посиланням на
   знахідку-основу), плюс `next_probes` — куди копати далі.
6. **document** — внести доведене й породжене в `DOCUMENT.md`: не дописати в хвіст, а вплести —
   документ тримається як зв'язний поточний стан думки, не лог.
7. **write** — оновити `LEDGER.md` / `SOURCES.md` / `IDEAS.md`, повернути JSON.

Ключове ребро — **зворотне**: знахідки verify стають сідом наступного diverge. Саме звідси свіжість —
ідеї виростають із того, що повернув пошук, а не з пріорів.

**Правило свіжості:** прогін без жодної `born_from: search` ідеї позначається сухим; два сухі
поспіль — агент прямо каже, що тема з цього кута вичерпана, і пропонує зміну кута [П].

**Схема виходу (JSON):**

```json
{
  "hypotheses": [ { "text": "...", "status": "confirmed|refuted|partial|open", "source": "url|null" } ],
  "fresh_ideas": [ { "text": "...", "born_from": "search|prior", "basis": "що саме зі знайденого" } ],
  "next_probes": [ "..." ],
  "document_delta": "один рядок: що змінилось у DOCUMENT.md",
  "dry_run": false
}
```

## Ledger — журнал сесії

Ledger — робочий журнал брейншторму: список усіх гіпотез сесії з поточним статусом кожної.
Гіпотеза народжується як `open`; після перевірки стає `confirmed` / `refuted` / `partial`, і до
неї чіпляється джерело. Поруч живуть ідеї, що виросли зі знахідок. Термін з бухгалтерії: головна
книга, де записано кожну операцію і завжди видно поточний стан рахунків — тут рахунки це гіпотези.

Три ролі:

- **Пам'ять між прогонами.** Агент стартує кожен прогін з чистим контекстом; `LEDGER.md` — єдине
  джерело того, що вже перевірено, тож те саме не ганяється двічі.
- **Напрям.** Спростоване закривається, підтверджене стає фундаментом наступного diverge — сесія
  рухається вперед, а не топчеться.
- **Прозорість.** Права панель TUI показує саме ledger: видно, що агент напрацював, без читання
  його сирого виводу.

## Робочий документ — результат сесії

`DOCUMENT.md` — те, заради чого сесія існує: зв'язний текст про тему, куди агент вплітає
підтверджені гіпотези та ідеї, що вижили. Правила:

- **Стан, не лог.** Ledger веде історію; документ тримає поточний стан думки. Агент переписує
  і перебудовує його, а не дописує в хвіст.
- **Вагомі тези — з опорою.** Твердження, що спираються на перевірене, несуть джерело;
  неперевірені ідеї позначені як гіпотетичні.
- **Форму задає користувач.** Репліки в чаті — «зроби це списком рішень», «додай розділ
  ризиків» — стають сідами, і агент перебудовує документ.
- **Він лишається.** Після `/quit` тека сесії з документом нікуди не зникає — це deliverable.

Веде документ агент (крок document у циклі) [П].

## Означення агента

Повний канон агента — окремий файл `.claude/agents/brainstormer.md`: хто він, цінності, цикл,
формати файлів workspace, контракт виходу, межі. Frontmatter:

```yaml
---
name: brainstormer
description: Ideation loop over a topic — hypothesize, verify on the live web, derive fresh
  ideas traceable to sources, maintain the working document (the session deliverable).
tools: WebSearch, WebFetch, Read, Write
model: opus
---
```

---

## Інтерфейс

- `/b <тема>` — старт: створює workspace, пише `TOPIC.md`, запускає перший прогін.
- Звичайні репліки — Haiku відповідає; те, що по темі, стає сідом наступного прогону.
- `/go` — примусовий прогін. Авто-тригер: після змістовної репліки по темі, якщо агент вільний
  і бюджет дозволяє [П].
- `/ledger` — поточний стан гіпотез та ідей.
- `/quit` — вихід; тека сесії з документом лишається.

Подача результатів: коли прогін завершився, Haiku вплітає підсумок у розмову — «поки ти думав,
я перевірила дві гіпотези: перша не тримається (джерело), зате з другої виросло ось що». Сирий
ledger — тільки на `/ledger` [П].

## Пісочниця та ізоляція виклику

Два яруси, кожен має рівно стільки прав, скільки потребує.

**Ярус 1 — Haiku (чат-мозок): рівно один тул.** Haiku — не Claude Code, а модель по HTTP API.
Єдина зовнішня дія, яку вона може ініціювати, — запустити прогін агента через тул
`run_brainstorm(seed)`. Більше tools нема: ні файлової системи, ні Bash, ні власного веб-пошуку,
ні спавну будь-чого іншого. Читання `DOCUMENT.md` / `LEDGER.md` для панелей, запуск `claude -p`,
парсинг JSON — це детермінований Python TUI, не рішення моделі. Вивід агента приходить до Haiku
як **дані через код**, а не як інструмент, яким вона керує.

**Ярус 2 — агент (brainstormer): замкнена четвірка.** Прогін іде з `cwd = тека сесії`; дозволені
тільки `WebSearch`, `WebFetch`, `Read`, `Write`, і Read/Write — лише в межах теки сесії, не
всього `MURARI_HOME`. Ніяких `../`, абсолютних шляхів чи симлінків назовні. Bash заборонений.
З білого списку викинуте все, що дозволяє спавнити ще один агент (Task / delegate / вкладені
сабагенти) — тож **ланцюг завжди рівно один рівень**: murari → brainstormer → результат, і жодна
третя ланка неможлива.

Технічно: `claude -p --agents brainstormer --allowedTools WebSearch,WebFetch,Read,Write
--disallowedTools Bash,Task --max-turns N`, запущений з `cwd` теки сесії; ту саму політику
дублюємо в `.claude/settings.json` теки, щоб вона трималась незалежно від того, як стартанули
процес [П].

**Дані vs інструкції.** Вивід агента для Haiku — дані, не інструкції: агент читав живий веб,
injection може приїхати через його синтез, тож Haiku подає результати як цитований матеріал і
не виконує вказівок з нього.

**Персональні дані.** Агент бачить лише тему, ledger і сіди цієї сесії; пам'яті між сесіями нема,
текти нічому. У пошукові запити йде тільки тема і зміст гіпотез.

## Бюджети (config)

| параметр | зміст | дефолт |
|---|---|---|
| `MURARI_RUNS` | прогонів агента на сесію | 6 |
| `MURARI_MAX_TURNS` | `--max-turns` одного прогону | 15 |
| `MURARI_MODEL` | модель агента | opus |
| `MURARI_HOME` | базова тека сесій | `~/murari` |

Opus 4.8 — дорога модель, тож капи тут не декор: `MURARI_RUNS` і `MURARI_MAX_TURNS` — основна
стеля витрат сесії.

---

## Definition of done

Сесія від `/b <тема>`: за 2–3 прогони ledger містить не менше чотирьох гіпотез із вердиктами та
джерелами і не менше двох свіжих ідей, простежуваних до конкретних знахідок; `DOCUMENT.md`
читається як зв'язний текст, вагомі тези мають джерела; репліка користувача помітно змінює напрям
наступного прогону і форму документа; сухі прогони чесно позначені; після `/quit` тека сесії з
документом лишилась, а нова сесія стартує з чистого аркуша.

## За дужками (перенесення в Lumi — потім)

Коли агент доведе себе тут: у Lumi він стає тулом у reply-loop або окремим burst worker'ом,
ledger переїжджає в її файлову пісочницю, повернення результатів іде через nudge/thought-механізм,
деідентифікація на границі тулу (правило WEB_SEARCH), а фінальний take — її, в її голосі
(правило Curiosity: reaction, not rebrief). Усе це свідомо поза прототипом.

## Відкриті питання

- Тригер-політика: авто після кожної репліки по темі чи лише `/go`.
- Формат подачі: Haiku переказує завжди, чи довгі результати — сирим блоком.
- Чи може користувач правити `DOCUMENT.md` напряму зовнішнім редактором (конфлікт записів з
  агентом), чи лише через чат.
- Розкладка трьох панелей: чат зліва + правий стовп, чи три колонки.
- Чи потрібна явна команда `/steer <напрям>` замість неявного сіда з реплік.
