# Ролі та стилі брейнштормінгу — murari

> **Ухвалений дизайн (2026-07-05).** Первинні джерела: [brainstormer.md](brainstormer.md),
> [architecture.md](architecture.md), [mission.md](mission.md). Усі рішення нижче — ✅ ухвалені
> (перелік — у таблиці наприкінці); історія обговорення — у git.

## Модель: хід → роль → стиль ✅

Три рівні:

1. **Хід (move)** — атомарна дія над станом сесії (LEDGER / IDEAS / SOURCES / DOCUMENT).
   Шість ходів: `generate`, `evaluate`, `deepen`, `oppose`, `mutate`, `weave`.
2. **Роль** — виконавець ходу. Роль ↔ хід 1:1. Роль може виконувати **агент або користувач** —
   людина є учасником брейншторму, а не глядачем.
3. **Стиль** — сценарій сесії: впорядкована послідовність ходів (з цілями). Стиль визначає,
   *які кроки і в якому порядку* відбуваються.

Координує все **Ведучий** — Haiku-чат (не одна з шести ролей): він визначає, яку роль зараз
грає користувач, обирає стиль і «закриває» агентами решту ролей, потрібних стилю.

## Шість ролей ✅

| Ключ | Роль | Хід | Tools |
|---|---|---|---|
| `generate` | **Фантазер** | породжує ідеї, навіть божевільні | без web |
| `evaluate` | **Суддя** | дає вердикти з джерелами | web |
| `deepen` | **Дослідник** | глибоко копає одну ідею | web |
| `oppose` | **Опонент** | шукає аргументи проти однієї ідеї | web |
| `mutate` | **Алхімік** | перетворює ідею через «а що якщо…» | web лише для `analogy` |
| `weave` | **Ткач** | вплітає стан у DOCUMENT.md | без web |

(Ключі — англійською в коді/контракті; імена — українською в чаті/документах.
«Ткач» — від канонічного «не дописуй у хвіст — вплітай».)

### `generate` — Фантазер
- **Суть:** дивергентне мислення; кількість і сміливість, не перевірність.
- **Вхід:** тема + LEDGER (щоб не повторювати закрите).
- **Вихід:** нові кандидати `open` у LEDGER; в IDEAS `born_from: prior`. **Ніколи не `confirmed`.**
- **Бюджет:** дешево, web низький/0.

### `evaluate` — Суддя *(ядро, доведене у v0.0)*
- **Суть:** конвергентне; відсіяти перевірні гіпотези й дати вердикт.
- **Вхід:** `open`-гіпотези з фактичним ядром.
- **Вихід:** `confirmed`/`refuted`/`partial` + url у LEDGER; рядки в SOURCES.
- **Бюджет:** середній, вшир.

### `deepen` — Дослідник
- **Суть:** вертикальне копання в **одну** ідею: факти, цифри, нюанси, умови.
- **Вхід:** одна обрана ідея (за id).
- **Вихід:** збагачений запис ідеї + кілька джерел; матеріал для окремого розділу DOCUMENT.
- **Бюджет:** дорого, web важкий. (Свідомо ламає «ощадність».)

### `oppose` — Опонент
- **Суть:** red-team; навмисний пошук аргументів **проти** однієї ідеї.
- **Вхід:** одна обрана (зазвичай `confirmed`, що виглядає однобоко).
- **Вихід:** контраргументи з джерелами; можливий зсув `confirmed → partial/refuted`.
- **Без переможця** ✅: мета Опонента — не «виграти суперечку», а **видобути й записати
  аргументи**. Продукт — зафіксовані контраргументи в LEDGER; статус зсувається лише від
  доказів, «перемога» не оголошується.
- **Випробуваність** ✅: якщо після атаки Опонента статус ідеї встояв, у LEDGER додається
  позначка `— випробувано: N` (скільки атак пережила). Випробувана `confirmed` вагоміша за просто `confirmed`.
- **Бюджет:** середній, точковий. (Пряме втілення «ворога Мури».)

### `mutate` — Алхімік
- **Суть:** латеральне; перетворити ідею навмисним «а що якщо» одного з **п'яти типів**.
- **Вхід:** одна ідея (для `combine` — дві) + тип мутації (обирає оркестратор випадково — див. нижче).
- **Вихід:** ідея-нащадок `open` з родоводом (`parents`); в IDEAS `born_from: mutation` + тип.
  **Ніколи не `confirmed`.**
- **Бюджет:** дешево, web низький/0.

**П'ять типів мутації** ✅:

| Тип | «А що якщо…» | Приклад перетворення |
|---|---|---|
| `scale` — масштаб | …×100 / ÷100? | ціна, швидкість, розмір, горизонт часу — до крайнощів в обидва боки |
| `invert` — інверсія | …навпаки? | обернути напрям, роль, мету, ключове припущення на протилежне |
| `transfer` — перенесення | …в іншому контексті? | інша галузь, країна, масштаб організації, епоха |
| `combine` — схрещення | …поєднати з іншою ідеєю? | злити **дві** ідеї в одну (SCAMPER «Combine»); нащадок має двох батьків |
| `analogy` — аналогія | …як це розв'язують деінде? | знайти, як подібну проблему розв'язала зовсім інша галузь, і перенести розв'язок |

`combine` — єдиний тип із двома входами: другу ідею обирає оркестратор (випадково або
найсильнішу за вердиктом). `analogy` може потребувати легкого web-пошуку (як розв'язують
у галузі-донорі) — виняток з «web низький/0».

**Випадковість живе в оркестраторі, не в моделі** ✅. LLM — поганий генератор випадковості
(системно тяжіє до «цікавого»); тип мутації (і за бажання — слово-стимул) обирає детермінований
Python (`random`) і передає в сід прогону. Агент *застосовує* тип, а не обирає його.

### `weave` — Ткач
- **Суть:** синтез; переписати DOCUMENT.md як зв'язний поточний стан думки.
- **Вхід:** увесь стан сесії (LEDGER/IDEAS/SOURCES).
- **Вихід:** перебудований DOCUMENT.md; вагомі тези — з джерелами, неперевірене — позначене.
- **Бюджет:** середньо, web 0.
- **Власність** ✅: **тільки Ткач пише DOCUMENT.md.** Решта ролей пишуть LEDGER/IDEAS/SOURCES.
  Це знімає конфлікт запису при кількох агентах і чітко відповідає, хто веде документ.
- **Користувач не редагує DOCUMENT.md напряму (v0)** ✅: його «перепиши висновок так…» —
  це *замовлення Ткачу* через чат, а не пряма правка файлу. (Закриває відкрите питання roadmap v0.3.)

## Користувач як учасник ✅

Користувач — сьомий гравець, який може зайняти будь-яку роль (окрім прямого запису
DOCUMENT.md — див. вище). **Ведучий (Haiku)** з реплік визначає, у якій ролі зараз людина,
і закриває агентами решту ролей, потрібних стилю:

| Репліка користувача | Роль |
|---|---|
| «а ще можна X, Y, Z…» | Фантазер |
| «ось стаття/факт/цифра…» | Дослідник (приносить матеріал) |
| «це не спрацює, бо…» | Опонент |
| «а що якщо навпаки / ×100 / в іншій галузі?» | Алхімік |
| «перевір оце твердження» | Суддя (замовляє вердикт) |
| «перепиши висновок так…» | замовлення Ткачу (не пряма правка) |

Принципи:
- **Компліментарність (дефолт):** агенти не дублюють роль користувача — грають решту.
  Якщо людина активно опонує, прогін `oppose` не потрібен — потрібні `deepen`/`evaluate`.
- **Адверсарна пара (у стилі «Суперечка»):** навпаки — агент навмисно стає в *протилежну*
  роль до користувача (людина захищає — агент атакує, і навпаки).
- **Провенанс** ✅: внески користувача маркуються: в IDEAS `born_from: user`; у журналі
  прогонів видно, хто виконав хід — агент чи людина.
- Внесок користувача, як і Фантазерів, — це `open`-кандидат: **вердикт усе одно вимагає джерела.**

## Стилі брейнштормінгу ✅ (усі шість)

Стиль = названа послідовність ходів (сценарій).
Скорочення: Ф=Фантазер, С=Суддя, Д=Дослідник, О=Опонент, А=Алхімік, Т=Ткач; `H` — обрана ідея.

| Ключ | Стиль | Суть | Приклад послідовності (6 ходів) |
|---|---|---|---|
| `explore` | **Фантазія вшир** | багато варіантів, широке поле | Ф → Ф → С → Ф → С → Т |
| `debate` | **Суперечка за один** | теза проти антитези над `H` | Д(H) → О(H) → Д(H) → О(H) → С(H) → Т |
| `riff` | **Фантазія вглиб одного** | розкручуємо один варіант | Д(H) → А(H) → Ф(довкола H) → А(H) → С → Т |
| `investigate` | **Розслідування** *(дефолт)* | ядро murari: гіпотези → перевірка | Ф → С → Д → С → О → Т |
| `evolve` | **Еволюція** | турнір поколінь: мутуй те, що вижило | Ф → С → А(вижилі) → С → А → Т |
| `premortem` | **Премортем** | «ідея вже провалилась — чому?» | О(H) → О(H) → Д(H) → С → Т |

Нотатки:
- `debate` — єдиний стиль з адверсарною парою людина↔агент (сторони можна мінятись).
  **Переможця немає** ✅: сторони — інструмент видобутку аргументів, не змагання; все, що
  видобуто обома сторонами, лягає в LEDGER, і саме це — результат суперечки.
- `evolve` спирається на родовід (`parents`): мутують лише ідеї з вердиктом `confirmed`/`partial`.
- `investigate` — те, що v0.0 вже довів; **стиль за замовчуванням**.
- Стиль обирає користувач явно (`/style`), або Ведучий за формулюванням теми; стиль можна змінити посеред сесії.
- **Стиль — шаблон, не рейки** ✅: Ведучий/scheduler може обґрунтовано відхилитись за станом LEDGER.

## Гейт джерел ✅

- **Генеративні ходи** (`generate`, `mutate`, внески користувача) → *кандидати*: статус `open`,
  `born_from: prior|mutation|user`. Ніколи не `confirmed`.
- **Оцінювальні ходи** (`evaluate`, `deepen`, `oppose`) → *докази*: вердикт + url.

«Джерело понад впевненість» тримається: генерація дає гіпотези, оцінка — вердикти.

## Ідентифікатори та родовід ✅

Ходи 3–5 працюють над однією ідеєю — на неї треба посилатись однозначно:

- Кожна гіпотеза отримує стабільний id: `H1`, `H2`, … (послідовно, без перевикористання).
- Формат: `- [H2][confirmed] текст — джерело: url — випробувано: 1`.
- Мутант: `- [H7][open] текст — parents: H3 — mutation: invert`.
- Схрещення (два батьки): `- [H9][open] текст — parents: H3+H5 — mutation: combine`.
- `target_idea` у контракті — id, не вільний текст.

Родовід дає **дерево ідей**: видно, з чого що виросло і які гілки вичерпані.

## Продуктивність ходу / `dry_run` per-move ✅

| Хід | Прогін «продуктивний», якщо… |
|---|---|
| generate | ≥3 нових кандидатів у LEDGER |
| evaluate | ≥1 вердикт із джерелом |
| deepen | ≥2 нові джерела/факти до обраної ідеї |
| oppose | ≥1 контраргумент із джерелом |
| mutate | ≥1 ідея-нащадок |
| weave | DOCUMENT.md перебудовано (не дописано в хвіст) |

(Пороги стартові, тюнити за практикою.)

**Журнал прогонів** ✅ у LEDGER — історія ходів для Ведучого/scheduler:

```
## Прогони
- 1: generate(агент) → H1..H5
- 2: evaluate(агент) → H2 confirmed, H4 refuted
- 3: oppose(користувач) → H2 контраргумент
- 4: mutate(invert, H2, агент) → H7
```

Лічильник сухості лишається один, але «сухість» прогону визначається критерієм його ходу.

|---|---|---|---|

## Реалізація ✅

**Шість рольових агентів замість одного** (напрям задано користувачем). Шлях впровадження —
core-outward:

1. **v0.1:** ролі як **параметризовані модулі одного канону** — прогін отримує `role`
   (+`target_idea`, `mutation_type`) у сіді; промпт кожної ролі — окремий модуль.
2. **Пізніше:** винести кожну роль в окремий canon-файл (`.claude/agents/…`) — крок малий,
   бо модулі вже розділені.

Tool-політика per роль ✅ (оркестратор звужує `--allowedTools` на прогін):

| Роль | Tools |
|---|---|
| Фантазер, Ткач | `Read, Write` |
| Алхімік | `Read, Write` (+`WebSearch` лише для `analogy`) |
| Суддя, Дослідник, Опонент | `WebSearch, WebFetch, Read, Write` |

Інваріанти тримаються:
- **Один рівень делегування:** агенти не кличуть одне одного. Оркестратор (детермінований
  Python) виконує стиль хід за ходом: `murari → роль → результат`, глибина завжди 1.
  Це fan-out на рівні 1, не рекурсія.
- **Спільний стан** — файли сесії; координація тільки через них.
- **Власність DOCUMENT.md** — тільки Ткач.
- **Ведучий (Haiku)** так само має рівно один інструмент; сигнатура розширюється:
  `run_brainstorm(seed, role, target_idea?, mutation_type?, style_step?)`.

## Вихідний контракт v2 ✅

```json
{
  "role": "generate|evaluate|deepen|oppose|mutate|weave",
  "target_idea": "H3 | null",
  "mutation_type": "scale|invert|transfer|combine|analogy | null",
  "hypotheses": [ { "id": "H7", "text": "...", "status": "confirmed|refuted|partial|open",
                    "source": "url|null", "parents": ["H3"] } ],
  "fresh_ideas": [ { "text": "...", "born_from": "search|prior|mutation|user", "basis": "..." } ],
  "next_probes": [ "..." ],
  "next_role": "запропонований наступний хід | null",
  "document_delta": "один рядок | null (не-weave ходи документ не чіпають)",
  "dry_run": false
}
```

Зміни проти v1 (v0.0): `role`, `target_idea` (id), `mutation_type`, `next_role`;
гіпотези з `id`/`parents` (1 батько для мутацій, 2 для `combine`, нема — корінна);
`born_from` розширено (`mutation`, `user`); `document_delta` nullable (документ пише лише Ткач).

**Перехід:** v0.0 запінив контракт v1 (тести/фікстури лишаються дійсними як v1);
v0.1 розширює схему до v2 і перепінює контрактні тести.
